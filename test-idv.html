02

Card Issuance — How Keys Are Born

This happens once at city hall when the card is issued. J-LIS generates the key pair inside the chip, registers the public key, and issues a signed X.509 certificate. The private key is generated on-chip and never exported.

1

Applicant visits municipal office

User presents existing ID. Staff verify physical identity. This is the root of trust — a real human was verified in person.

2

Key pair generated inside the chip

The IC chip's secure element generates an EC P-256 key pair on-device. Private key is written to tamper-resistant storage and cannot be exported.

// Conceptual — actual key gen is internal to chip OS
Chip generates: privateKey_auth → secure element storage
Chip exports: publicKey_auth → CSR to J-LIS
3

J-LIS issues X.509 certificate

J-LIS signs the public key with their Intermediate CA private key, creating an X.509 certificate that binds the key to the person's identity.

// Certificate structure stored on chip
Subject: 鈴木太郎, 1990-01-01, 東京都...
PublicKey: EC P-256 [publicKey_auth]
Issuer: J-LIS Auth CA (Intermediate)
Signature: Sign(hash(above), privateKey_JLIS_int) ✅
Valid: 2022-01-15 → 2027-01-14 (5 years)
4

Certificate written back to chip

The signed certificate is stored in the JPKI applet's Elementary File (EF). User sets their 4-digit auth PIN. Card is ready.

04

Live Verification — NFC APDU Commands

This happens every time a user verifies their identity in your app. The gateway SDK sends ISO 7816 APDU commands over NFC to the chip. Each hex byte has a specific meaning.

App → Chip (command)
Chip → App (response)
Warning state
1

Select JPKI Applet

Tell the chip we want to talk to the JPKI application specifically. The chip contains multiple applets — this targets the right one.

→ App sends:
00 A4 04 0C 0A D3 92 F0 00 26 01 00 00 00 01
00CLA — standard class A4INS — SELECT command 04 0CP1/P2 — select by AID, no response data D392...0001AID — JPKI application identifier
← Chip responds:
90 00 // Success
2

Select Auth Certificate File

Navigate to the Elementary File (EF) that contains the authentication certificate. File ID 000A is the auth cert.

→ App sends:
00 A4 02 0C 02 00 0A
A4INS — SELECT 02P1 — select by file ID 00 0AFile ID — auth certificate EF
← Chip responds:
90 00
3

Read the X.509 Certificate

Read the DER-encoded certificate from the chip. This is the public cert — no PIN needed. App parses it to get the public key and identity claims.

→ App sends:
00 B0 00 00 00
B0INS — READ BINARY 00 00P1/P2 — offset 0 (read from start) 00Le — read all available bytes
← Chip responds:
[DER-encoded X.509 certificate bytes] 90 00
// App extracts: publicKey, subject name, DOB, issuer, validity dates
4

Select Auth Private Key File

Navigate to the key reference for authentication. This tells the chip which private key to use for the upcoming signing operation.

→ App sends:
00 A4 02 0C 02 00 17
00 17File ID — auth private key reference
← Chip responds:
90 00
5

VERIFY PIN — Security Boundary

User enters their 4-digit PIN. This is the gatekeeper — the chip will not sign anything until the correct PIN is presented. Wrong PIN reduces remaining attempts.

→ App sends:
00 20 00 80 04 [31 32 33 34]
20INS — VERIFY 00 80P1/P2 — auth PIN reference 04Lc — 4 bytes of PIN data 31 32 33 34PIN "1234" as ASCII bytes
← Chip responds:
90 00 // Correct PIN ✅
63 C2 // Wrong PIN — 2 attempts remaining ⚠️
63 C1 // Wrong PIN — 1 attempt remaining 🚨
69 83 // Card LOCKED — visit city hall 🔒
6

INTERNAL AUTHENTICATE — Challenge / Response

App sends a cryptographically random 32-byte challenge. The chip signs it with the private key internally. The signature proves the chip holds the private key — without ever exposing it.

→ App sends:
00 88 00 00 20 [A3 F2 91 ... 32 random bytes] 00
88INS — INTERNAL AUTHENTICATE 20Lc — 32 bytes of challenge data [32 bytes]Random challenge (never reused) 00Le — return all response bytes
← Chip responds:
[64-byte ECDSA signature] 90 00
// Chip: signature = Sign(challenge, privateKey_auth)
// Private key never left the chip
7

Verify Signature + OCSP Check

Gateway verifies the signature using the public key from the cert. Then checks with J-LIS OCSP that the cert hasn't been revoked.

// Gateway / your backend
verify(signature, challenge, publicKey_from_cert)
→ signature valid ✅ (person holds the private key)

POST https://ocsp.jpki.go.jp/ [cert serial number]
→ status: "good" ✅ (cert not revoked)

Identity confirmed. Issue JWT.

Why revocation matters: if a card is lost/stolen, the owner reports it → J-LIS revokes the cert → OCSP returns "revoked" → verification fails even if attacker has the physical card and knows the PIN. Cost: ¥2 per check.

What Your Backend Receives

Gateway returns a signed JWT. Your backend validates it and creates the user record.

{ "iss": "https://gateway.example", "sub": "jpki|a3f9c821b4e0", "assurance_level": "IAL2", "verification_method": "jpki_nfc", "jpki": { "cert_serial": "0A3F92B1C4D5", "cert_valid_until": "2027-01-14", "ocsp_status": "good" }, "identity": { "name": "鈴木 太郎", "birthdate": "1990-01-01", "address": "東京都千代田区...", "gender": "male" } }

Key fields and what to do with them:

cert_serial
Use as durable identity key. Same person = same serial even if name/address changes.
assurance_level: IAL2
Government-grade. Equivalent to in-person identity verification.
cert_valid_until
Store this. Drive re-verification flow when cert approaches expiry (5yr cycle).
ocsp_status: "good"
Always assert this before creating user. "revoked" = reject immediately.
